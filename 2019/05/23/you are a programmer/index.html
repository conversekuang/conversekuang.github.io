¶you are a programmer

​ 之前网上总是笑称程序员和产品经理之间的什么仇什么怨，虽然看到眼里，但是并没有经历。最近在做项目，对于提需求，改需求这方面，真的深有体会。以及买家秀与买家秀诸如此类的问题。一个项目感觉需求就没搞清楚过，再加上开发产品在不断的在变形，都快上线可能又面临各种问题。

​ 首先，对于软件外包这件事情上，会在项目成功率大打折扣。公司没有自己的核心研发团队依靠外部来做，首先，沟通成本的增加。每次需要通过，消息、电话的方式而非当年进行沟通，效率和理解层面都欠佳。次之，产品设计与阶段性开发成果的差别，也就是买家秀和卖家秀的差距。设计航空母舰，可能是个大轮船，或者说看着像航母但是功能不齐全。一问，这儿没有做好，再问，那个不是核心功能，一而再再而三的大打折扣，变成后续迭代更新。所以，在成品方面具有太大的不可控性。此外，由于研发与设计的不断偏离，给用户或者使用者带来太多的不便，但是最后变成使用者迁就系统。

​ 本次项目中，该三点均体会深刻，尤其是第三点。作为一个程序员，难道不应该常思考用户是否操作方面，用户可能需要什么功能呢？今天，我就被告知要用他们开发的系统进行数据录入。后台并没有导入操作，没有批量操作。此外，地图数据的调节竟然不能在后台进行。还需要一遍调整一遍放到APP端进行观察。一旦要调节数据，对所有用户在APP上都是可见状态，显然该操作不科学。百度地图的线路信息都是用点坐标连接，我爬过一个公交车轨迹，有262个坐标点，我也是采用描点法进行显示线路方向。但是呢，本项目APP开发人员端采用的是两点之间调取坐标的方式，需要少量的精确坐标，要不断的调整的坐标否则会根据有各种不可思议的拐弯和交叉。就在这个方面，浪费过太多的时间。明明都已经整理好数据可以直接显示，非要采用规划方式。后续后台操作人员取点线路坐标容易，还是不断尝试规划方式方面。各有各的优势，我承认。但是不应该保持一致么？？ps，之前用的高德，后面突然告诉你他们用的baidu，又浪费时间把所有信息调整加显示。太多的无用功了。

​ 说说为什么名字叫做“你是一个程序员”，如上所说，一个程序员让我通过后台把所有数据加入进去，马上上线了，为什么你不写个sql直接把数据导入？非要让人一个个点？虽然只有120条数据，每个数据4个字段。算一下，需要切换excel文档，复制ctrl+c，粘贴到系统ctrl+v。一条数据有多少秒？ 我是老年人姑且算一条数据20s吧，120*20 = 2400s，因为本来数据就不全，这还不算补全数据需要的时间，以及乱七八糟的时间，瞅着那么多数据万一输入错怎么办？检查一遍？成本太高了。主要是录入这个数据才是第一步，后面还需要再将这个120数据进行组合。再找一遍。想想就觉得整个人都不好了。作为会撸代码的人，我不能忍受这样低效的无意义工作。因此，怎么样快速提交？是我第一个想到的。毕竟我是拒绝一个个傻瓜式的输入系统。

​ 下面具体说一下，我如何面对的？

  1. 观察添加数据页的字段与数据。发现一个字段没有，需要通过人为识别，每条都要对行政区域进行判断，对于不熟悉的人必然需要查，然后一个个录入。我采用百度API的方式，通过关键字发送请求，在对比请求结果中找到对应地点的行政区域。复习了requests库，并练习pandas进行写入csv与xlsx。

  2. 其中发现一个问题：我是将xlsx文件中的数据读取出来当做参数去百度请求，响应的结果中的key要与该数据进行判断。问题就出现了，print 汉字之后，并没有异常，但是在请求结果中就是无法匹配，最后才发现原来是BOM搞的鬼。’ \xEF\xBB\xBF’。每行第一列的数据都会带有这个东西，但是print显示出来就一样。因此要把该头删除。因为，后续对数据库查找与匹配也存在问题，之前就被坑过。通过以下方法进行消除。

  3. 数据准备好了，要怎么提交上去？真的要一个个点？我内心拒绝的，好歹是个撸代码的，我宁愿用代码解决重复的劳动，最起码我练习了各种操作也比ctrl cv强。好歹也是爬虫出来的，大体思路先按请求试一下，实在不行我们可以用selenium模拟操作嘛，虽然慢一点。所以首先，先按第一个思路。打开抓包软件，先试着提交数据。看到了提交的request。除了cookie和header要处理一下，提交数据在body而不是表单提交。应该用requests库的post就能搞定。cookie就用浏览器刚才生成的，这样就避免登录了。完全没必要在去处理登录。直接用sessionid带到cookie中。在爬虫中已经是轻车熟路了。

    其中，data是在body中的字段，之前差点以为是json，结果前几次直接json = json.dumps(res),全都倒name字段中了。结果当然是请求

  4. 下面是sql返回的是错误，查了一下是自己定义的字段和他们数据库定义的不一样。最后核对字段名称，类型。第一次将数据类型没有变成str，插入成功但是有些字段并没有显示出来。最后所有字段转换成str类型。

  5. 结果，当然是成功了。因为成功请求成功后sleep(1)，一秒钟。所以140s后完成了。数据全都灌入成功。数据录入成功。虽然花的时间有点多，但是在准确率上应该是没问题的，最起码不对串行。其次，虽然只有120条数据但是，可以具有可用性。同样能够处理1200条数据，这样就不是手工录入能够比拟的了。最后，还是发现并且解决一些问题。增加了实战了操作多个常用库。

​ 所以值得庆幸的是，我是一个程序员。我学习的东西能够为我减轻一定的工作量。让我庆幸自己掌握一点东西能够在实际中应用到。还好我之前学到这些东西。
